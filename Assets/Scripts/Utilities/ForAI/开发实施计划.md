# MobileGameFinal 开发实施计划

## 1. 文档用途

本文档用于指导后续 AI 或开发者按照小步、可验证、可独立提交的方式，实现一个轻量级《杀戮尖塔》式卡牌 Roguelike 游戏。

本文档只描述开发计划，不代表所有内容都需要立即实现。后续每次开发只能选择一个阶段，完成验证后停止，等待开发者确认再进入下一阶段。

### 1.1 核心目标

- 使用 ScriptableObject 承载可配置的游戏数据。
- 使用事件系统解耦战斗、卡牌、遗物、Buff 和 UI。
- 使用 Manager 编排关键业务流程。
- 实现玩家回合、卡牌战斗、敌人行动和地图探索的最小完整循环。
- 控制课程作业规模，避免为了架构完整性引入不必要的复杂度。

### 1.2 开发规则

1. 每次只实现本文档中的一个阶段。
2. 每个阶段完成后必须保证项目能够编译。
3. 不提前创建尚未使用的大量抽象接口。
4. 不引入依赖注入框架、Command Dispatcher、ECS 或复杂领域框架。
5. 不为了未来可能出现的需求提前实现网络同步、回放、撤销等功能。
6. AI 不自动创建 Git 提交，除非开发者明确要求。
7. 不修改提交时间、作者或其他历史信息，不伪造 Git 历史。
8. 已有用户代码或资源不得擅自覆盖；执行阶段前先检查工作区状态。

## 2. 架构原则

### 2.1 总体结构

项目采用以下简单结构：

```text
ScriptableObject 数据
        ↓
Manager 业务编排
        ↓
角色 / 卡牌 / 地图运行时对象
        ↓
Event 通知 Buff、遗物、UI 和表现层
```

各层职责如下：

| 层级 | 职责 | 示例 |
|---|---|---|
| 数据层 | 保存可配置静态数据 | CardDataSO、EnemyDataSO、MapConfigSO |
| 业务层 | 明确执行游戏规则和流程 | TurnManager、BattleManager、CardManager |
| 运行时模型 | 保存本局动态状态 | Character、CardDeck、BattleCard |
| 事件层 | 通知已发生的事实，编排额外效果 | TurnStartedEvent、BeforeDamageEvent |
| 表现层 | UI、动画、音效和输入 | BattlePanel、CardView、HealthBar |

### 2.2 Event 的定位

事件只表达已经发生或正在结算的游戏事实，不用于命令业务系统执行任意操作。

正确示例：

- `PlayerTurnStartedEvent`
- `CardPlayedEvent`
- `BeforeDamageEvent`
- `DamageResolvedEvent`
- `CharacterDiedEvent`

不推荐的事件：

- `ChangeTurnToEvent`
- `SetPlayerHealthEvent`
- `ExecuteEnemyActionEvent`

后面这些名称表达的是命令，应由 UI 或业务代码直接调用对应 Manager 的公开方法。

### 2.3 UI 与业务系统的交互

UI 直接调用语义明确的公开方法：

```text
StartBattleAsync()
EndPlayerTurnAsync()
TryPlayCardAsync(card, target)
SelectRoom(room)
```

业务完成后再发布 Event，UI 也可以监听 Event 更新显示。

```text
UI 输入
  ↓ 直接调用
Manager
  ↓ 修改游戏状态
Event
  ├─ Buff
  ├─ 遗物
  ├─ UI
  └─ 音效/动画
```

### 2.4 回合流程

回合系统不使用持续运行的 `while` Loop，也不在玩家操作阶段挂起一个等待任务。

采用分段时序：

```text
开始战斗
  → 玩家回合开始
  → 进入玩家出牌阶段
  → 方法执行结束，等待用户操作

用户点击结束回合
  → 玩家回合结束
  → 敌人回合开始
  → 敌人依次行动
  → 敌人回合结束
  → 下一次玩家回合开始
  → 再次停在玩家出牌阶段
```

本项目暂不为回合系统使用 `CancellationToken`。场景或战斗对象销毁后，不应存在持续等待的回合任务。

### 2.5 Priority 规则

事件系统支持 Priority，用于 Buff、遗物和表现效果的结算顺序。

统一约定：

- Priority 数字越小，越早执行。
- 相同 Priority 按注册顺序执行。
- 异步监听器逐个 `await`，不得并行执行。
- 核心规则不能依赖相同 Priority 的注册顺序。

建议区间：

| Priority | 用途 |
|---:|---|
| 0-99 | 系统强制规则 |
| 100-199 | 状态效果、Buff |
| 200-299 | 遗物 |
| 300-399 | 统计、任务、日志 |
| 400-499 | UI、动画、音效 |

以上区间只是项目约定，不需要创建复杂的动态优先级系统。

## 3. 建议目录结构

目录只在对应阶段开始时创建，不要求第一阶段一次性建立所有空目录。

```text
Assets/Scripts/
├── Core/
│   ├── Events/
│   ├── Variables/
│   └── Utilities/
├── Gameplay/
│   ├── Battle/
│   ├── Characters/
│   ├── Cards/
│   ├── Effects/
│   ├── Relics/
│   └── Map/
├── Presentation/
│   ├── Battle/
│   ├── Cards/
│   ├── Map/
│   └── Common/
└── Tests/
    ├── EditMode/
    └── PlayMode/
```

命名约定：

- ScriptableObject 数据类使用 `SO` 后缀，例如 `CardDataSO`。
- 事实事件使用 `Event` 后缀，例如 `CardPlayedEvent`。
- Manager 只用于真正负责编排多个对象的类。
- 纯运行时数据模型不要继承 MonoBehaviour。
- 文件名与公开类型名保持一致。

## 阶段 2：Priority 事件核心

### 目标

实现基于 ScriptableObject 的泛型事件通道，并支持同步和异步监听器按 Priority 顺序执行。

### 建议文件

```text
Core/Events/BaseEventSO.cs
Core/Events/EventListenerEntry.cs
```

### 必要能力

- 注册同步监听器。
- 注册异步监听器。
- 注销监听器。
- 防止同一个监听器重复注册。
- Priority 越小越早执行。
- 同优先级按注册顺序执行。
- 发布过程中允许监听器注册或注销，因此发布时应使用快照。
- 监听器逐个执行，确保结算顺序稳定。
- 保存最后一次事件发送者，便于调试。

### API 方向

具体命名可根据实现调整，但语义应保持简单：

```csharp
Register(listener, priority);
Unregister(listener);
RaiseAsync(value, sender);
```

### 设计限制

- 事件系统只处理 Event，不处理 Command。
- 不实现事件继承分发。
- 不实现跨线程调用。
- 不实现全局字符串事件名。
- 不实现异常吞噬；监听器异常默认终止本次事件链，便于发现错误。

### 测试重点

1. Priority 顺序正确。
2. 相同 Priority 保持注册顺序。
3. 重复注册不会执行两次。
4. 注销后不再执行。
5. 发布期间注销不会破坏当前遍历。
6. 异步监听器会被完整等待。

### 完成标准

- 核心事件类能够编译。
- 至少有纯代码测试验证排序规则。
- 尚未创建任何战斗事件资产。

- 各事件通道沿用同一套 Priority 规则。
- 不复制事件核心逻辑。

## 阶段 9：玩家回合时序

### 目标

实现开始战斗到玩家可操作阶段的分段流程。

### 建议文件

```text
Gameplay/Battle/BattleManager.cs
Gameplay/Battle/TurnManager.cs
```

### 时序

```text
StartBattleAsync
  → 初始化战斗数据
  → 发布 BattleStartedEvent
  → StartPlayerTurnAsync
      → 恢复能量
      → 清除或结算回合状态
      → 抽满至 5 张牌（牌堆完成后接入）
      → 发布 PlayerTurnStartedEvent
      → 设置阶段为 PlayerAction
  → 方法结束
```

### 防重入

使用一个简单的 `_isTransitioning` 布尔值阻止：

- 重复开始战斗。
- 时序执行中重复点击结束回合。
- 两条敌人行动序列并行执行。

### 完成标准

- 开始战斗后方法自然结束。
- 当前阶段停在玩家操作阶段。
- 不存在 `while` Loop。
- 不存在等待玩家输入的 `UniTaskCompletionSource`。

## 阶段 10：敌人回合时序

### 目标

实现 `EndPlayerTurnAsync()`，顺序执行玩家结束和敌人回合。

### 时序

```text
EndPlayerTurnAsync
  → 验证当前为 PlayerAction
  → 设置 _isTransitioning
  → 玩家回合结束
      → 弃掉手牌
      → 发布 PlayerTurnEndedEvent
  → 敌人回合开始
      → 发布 EnemyTurnStartedEvent
  → 按场景顺序逐个执行存活敌人行动
  → 敌人回合结束
      → 发布 EnemyTurnEndedEvent
  → 若战斗未结束，开始下一玩家回合
  → 清除 _isTransitioning
```

### 完成标准

- 连续点击结束回合只触发一次。
- 多个敌人严格按列表顺序行动。
- 敌人行动完成后回到玩家阶段。

---

## 阶段 11：战斗胜负

### 目标

在关键结算点判断战斗是否结束，避免死亡后继续执行动作。

### 检查点

- 玩家受到伤害后。
- 敌人受到伤害后。
- 每张卡牌效果执行后。
- 每个敌人行动后。

### 结果

- 玩家生命值小于等于 0：失败。
- 全部敌人死亡：胜利。
- 发布一次 `BattleEndedEvent`。
- 后续时序不再继续。

### 完成标准

- 结果只产生一次。
- 最后一个敌人死亡后不会进入敌方回合。
- 玩家死亡后剩余敌人不再行动。

### 第一批效果

- 造成伤害。

- 易伤。

### 规则

### 完成标准

- 效果可以纯代码测试。

## 阶段 14：牌堆逻辑

### 目标

实现不依赖卡牌 GameObject 的纯数据牌堆。

### 建议文件

```text
Gameplay/Cards/CardDeck.cs
Gameplay/Cards/CardPile.cs
```

### 牌堆

- 抽牌堆。
- 手牌。
- 弃牌堆。

### 必要行为

- 消耗牌进入消耗堆。

### 完成标准

- 测试覆盖洗牌外的确定性规则。
- 测试可注入随机源，避免随机测试不稳定。

## 阶段 15：出牌流程

### 目标

完成玩家从选择卡牌到效果结算的业务流程。

### 建议 API

```text
TryPlayCardAsync(CardInstance card, CharacterModel target)
```

### 校验

- 当前必须为玩家操作阶段。
- 卡牌必须在手牌中。
- 能量足够。
- 目标类型正确。
- 战斗尚未结束。
- 当前没有另一张牌正在结算。

### 执行

1. 扣除能量。
2. 依次执行卡牌效果。
3. 发布 `CardPlayedEvent`。
4. 根据标签移入弃牌堆或消耗堆。
5. 检查战斗结果。

### 完成标准

- 重复点击同一张牌不会执行两次。
- 费用不足时不修改任何状态。
- 多效果严格按顺序执行。

## 阶段 16：Buff 与遗物

### 目标

建立最小的事件订阅生命周期，使 Buff 和遗物能够通过 Priority 插入结算。

### 建议结构

```text
Gameplay/Relics/RelicSO.cs
Gameplay/Relics/RelicRuntime.cs
Gameplay/Battle/Statuses/StatusEffect.cs
```

### 生命周期

- 获得遗物时注册监听器。
- 丢失或战斗结束时注销需要清理的监听器。
- Buff 根据持续时间在回合开始或结束时减少层数。

### 示例

- 战斗开始增加攻击力。
- 玩家首次受伤时抽牌。
- 每回合开始获得格挡。
- 攻击若干次后恢复能量。

### 完成标准

- Priority 顺序可以通过测试证明。
- 移除遗物后不会继续触发。
- 不为每个遗物创建独立 Manager。

### 目标

### 必要行为

## 阶段 20：表现层、资源和存档

### 目标

在核心规则稳定后接入 Unity 表现功能。

### 内容

- UI Toolkit 面板。
- 卡牌拖拽和目标选择。
- 手牌布局。
- DOTween 动画。
- 敌人意图显示。
- 血条和状态图标。
- Addressables 场景或资源加载。
- JSON 存档。

### 接入原则

- UI 不直接修改角色字段。
- UI 调用 Manager 的公开业务方法。
- UI 监听 Event 更新显示。
- 动画不决定最终战斗数值。
- 存档保存稳定 ID 和运行时状态，不直接序列化场景对象引用。

### 完成标准

- UI 与核心模型分离。
- 关闭动画后战斗规则仍可运行。
- 存档可恢复牌组、角色、地图和当前进度。

课程作业的第一版只需要以下内容：
